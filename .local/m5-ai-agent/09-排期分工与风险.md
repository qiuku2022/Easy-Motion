## 角色分工建议

| 角色 | 负责 Phase | 技术栈 |
|------|------------|--------|
| Node 主进程 | 0,1,2,3,5 | Agent、IPC、文件服务、generator 联动 |
| 前端 | 2,3,4,6 | conversationStore、AI 组件、状态联动 |
| LLM/Prompt | 3,4,6 | Prompt、工具 schema、10 条回归用例 |

Phase 1 与 Phase 2 必须由不同开发者并行，避免阻塞 Phase 3。

## 风险与预案

| 风险 | 概率 | 影响 | 预案 |
|------|------|------|------|
| LLM tool_use 不稳定 | 高 | 高 | JSON Patch 兜底；简化模式 |
| MiniMax 工具调用兼容 | 中 | 中 | 抽象 `LlmProvider`；备 OpenAI 协议测试 |
| Agent 与手动编辑冲突 | 中 | 中 | `lastModifiedBy` + 确认框 |
| 多模态成本高/不准 | 高 | 低 | Phase 6 降为 stretch；文字为主 |
| LangChain 引入诱惑 | 低 | 中 | M5 明确不引入；代码 review 把关 |
| API 成本 | 中 | 中 | 开发用低温度 + 短上下文；日志脱敏 |

## 成本与调试

- 开发环境 Key 放 `secrets.json`，禁止提交 `.env`
- Prompt 回归用固定 10 条，避免每次全量手动测
- 主进程 Agent 日志：`[agent]` 前缀，不打印完整 timeline JSON（过大）
