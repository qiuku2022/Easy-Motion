# 技术规格

## Remotion 版本策略
- **锁定版本**：Remotion v4.0.269（2026-05-29 最新稳定版）
- **版本管理**：子项目 `package.json` 中固定版本号（不带 `^`/`~`），`package-lock.json` 提交到版本控制
- **升级流程**：如需升级，先在测试子项目验证，通过后批量更新模板 `package.json`，再应用至新项目
- **多版本共存**：原则上统一版本；如需特殊版本，由总项目 `project.json` 的 `remotionVersion` 字段指定，子项目独立安装
- **不使用开发版/主分支**，确保稳定性
- 后续升级需经过充分测试

## 技术选型
- **GUI 框架**：Electron
- **前端 UI**：React + shadcn/ui + Tailwind CSS
- **动画引擎**：Remotion（官方发行版）
- **后端服务**：Node.js
## 数据分析模块
- **技术栈**：Python + FastAPI
- **主要职责**：
  - 网页爬取：抓取网页内容、数据、参考素材
  - 数据清洗与格式化：转换为结构化数据（JSON/CSV）
  - 本地存储：保存到项目 `data/` 目录
  - 为大模型提供结构化数据输入

## 爬虫模块
- **框架**：Playwright（模拟真实浏览器，支持动态页面渲染）
- **反爬策略**：
  - Playwright Stealth 模式（防检测）
  - 随机请求延迟（1-3 秒）
  - 常见浏览器 User-Agent 轮换
- **FastAPI 接口**：
  - `POST /crawl`：传入 URL，返回网页内容/结构化数据
  - `POST /crawl/save`：传入 URL 和目标路径，直接保存到 `data/` 目录
  - `POST /crawl/screenshot`：截图保存为图片参考
- **Agent 框架**：LangChain（用于构建多模态对话 Agent）
- **包管理器**：pnpm（全局存储节省空间，子项目依赖共享，安装速度快）

## LLM 与 Agent
- 基于 LangChain 构建对话 Agent
- 支持多模态模型（图片参考 + 文字描述）
- LLM 提供商可配置（OpenAI、Claude、本地模型等）
- Agent 负责：
  - 解析用户自然语言需求
  - 解析参考图片的视觉结构
  - 生成 Remotion 组件配置
  - 执行对话调整指令

## Electron 主进程 ↔ 渲染进程通信方案

### 通信模式
采用 **ipcMain / ipcRenderer + 共享文件** 双轨通信：
- **高频/实时通信**（预览控制、播放状态、对话消息）：通过 `ipcMain.invoke` / `ipcRenderer.send` 同步/异步通道
- **大数据/状态同步**（项目文件读写、素材传输）：通过 Node.js `fs` 主进程操作，渲染进程通过 IPC 请求数据
- **跨窗口通信**（主窗口 ↔ 项目窗口）：通过 `ipcMain` 中转广播

### IPC 消息命名规范
- `main:xxx`：渲染进程 → 主进程请求（如 `main:openProject`, `main:saveProject`）
- `renderer:xxx`：主进程 → 渲染进程通知（如 `renderer:projectLoaded`, `renderer:renderProgress`）

## 状态管理

### 选型：Zustand + Immer
- **Zustand**：轻量、无 Provider 嵌套、支持多 Store，适合 Electron 多窗口隔离
- **Immer**：不可变状态更新，简化复杂嵌套对象（时间线、关键帧）的修改逻辑
- **持久化**：项目相关 Store 自动序列化到 `project.json`，通过 IPC 触发主进程写盘

### Store 划分
- `projectStore`：总项目/子项目列表、当前激活子项目
- `timelineStore`：时间线状态（轨道、片段、播放头位置）
- `conversationStore`：对话历史、当前 Agent 任务状态
- `uiStore`：面板显隐、布局、选中元素
- `previewStore`：播放状态（播放/暂停/帧号）、预览缩放

## Remotion Player 集成方案

### 集成方式：iframe + 子项目 Dev Server
- 每个子项目的 Remotion 预览运行在**独立的 iframe** 中，iframe 的 `src` 指向子项目的本地 Remotion dev server（webpack 编译服务）
- 原因：Remotion Player 需要完整的 webpack/React 编译环境，无法仅通过 JSON 配置直接渲染；iframe 同时实现 DOM/CSS 隔离，避免与 Electron UI 冲突

### 预览更新流程
```
Electron Renderer (React UI)
  → timeline 修改（手动或 AI 生成）
  → 触发 Generator（防抖 500ms）
  → 生成 TSX 代码 → 写入 子项目/remotion/src/
  → 子项目 webpack dev server HMR 或 iframe reload
  → iframe 内 Remotion Player 渲染最新效果
```

- **代码生成**：每次 timeline 修改后，Generator 将 `subproject.json` 编译为 `Root.tsx` + `MainSequence.tsx` + Layer 组件，写入子项目的 `remotion/src/` 目录
- **实时重载**：子项目内运行 Remotion 的 webpack dev server，支持 HMR（热模块替换）
- **预览更新**：代码写入后，通知 iframe 执行 HMR；如 HMR 失败则自动 reload iframe
- **状态同步**：iframe 内的 Remotion Player 通过 `window.postMessage` 与父窗口双向通信
  - 父 → iframe：播放控制（`seekTo(frame)`、播放/暂停、设置缩放）
  - iframe → 父：播放头位置变化、用户点击元素（用于选中时间线片段）

### 通信链路
```
Electron Renderer (React UI)
  → ipcMain.invoke('preview:play') / invoke('preview:seek', frame)
  → 主进程 → iframe.postMessage({ type: 'PLAY' / 'SEEK', payload: frame })
  → Remotion Player 响应

iframe (Remotion Player)
  → window.postMessage({ type: 'FRAME_CHANGE', payload: currentFrame }, '*')
  → Electron Renderer 监听 → 更新 previewStore.currentFrame
```

## 数据存储方案

### 项目配置：JSON 文件
- `project.json` / `subproject.json`：主进程独占写，渲染进程只读（通过 IPC 请求修改）
- 写盘策略：修改后 500ms 防抖自动保存；关键操作（删除、重命名）立即同步

### 素材元数据：SQLite
- 本地 SQLite 数据库（`~/EasyMotion/library.db`）
- 存储：素材路径、类型、尺寸/时长、缩略图路径、导入时间、所属项目 ID
- 查询场景：素材库搜索、去重判断、项目素材统计

### 对话历史：JSON 文件
- 每个子项目 `conversation.json`，按对话轮次存储（用户输入、图片引用、AI 回复、代码 diff、执行状态）
- 支持按版本回滚（读取历史 `conversation.json` 恢复状态）

### 缩略图缓存：文件系统
- 素材导入时异步生成缩略图（图片：缩略图；视频：首帧截图），存于 `项目根目录/.thumbnails/`
- 命名规则：`{素材UUID}_{宽}x{高}.jpg`，避免重复生成

### 项目快照
- 存储路径：`项目根目录/timeline-snapshots/`
- 触发条件：撤销栈每累积 10 步自动生成一个快照，或在关键操作（如生成动画完成、手动保存）后生成
- 文件命名：`{timestamp}.json`（Unix 时间戳毫秒）
- 内容：完整的 `subproject.json` + `conversation.json` 快照
- 数量限制：最多保留 50 个快照，超出时自动删除最早的
- 崩溃恢复：启动时扫描该目录，若发现未正常关闭的项目，提示用户恢复到最近快照

## 项目文件并发控制

### 单写队列机制
- 主进程维护**项目文件写队列**，同一时刻只有一个写操作在执行
- 渲染进程的所有写请求（通过 IPC）进入队列，主进程串行处理
- 避免多窗口/多操作同时写 `project.json` 导致数据损坏

### 文件校验
- 每次写 `project.json` 前，先读取当前内容计算 SHA-256，与内存中的基准哈希比对
- 若发现外部修改（理论上不应发生），触发警告并让用户选择覆盖/合并
- 写入完成后立即验证 JSON 可解析性，失败时回退到备份文件（`project.json.bak`）

## 错误处理与降级策略

### 统一错误分层
| 层级 | 处理方 | 示例 | 用户反馈 |
|------|--------|------|----------|
| UI 错误 | 渲染进程 | 表单校验失败、拖拽到无效区域 | Toast 提示，不中断流程 |
| 业务错误 | 主进程 | 文件不存在、LLM 调用超时 | 弹窗提示，提供重试/跳过 |
| 系统错误 | 主进程 | 磁盘满、权限不足 | 弹窗 + 日志记录，引导用户解决 |
| 崩溃错误 | 主进程 | Electron 渲染进程崩溃、Python 子进程崩溃 | 自动恢复/重启，数据恢复提示 |

### 关键降级方案
- **LLM 调用失败**：首次失败自动重试 1 次；仍失败则提示用户检查网络/API Key，允许手动重试或跳过该步骤
- **Python 子进程崩溃**：主进程检测心跳（每 5 秒 ping），无响应则自动重启；重启期间爬虫/数据分析接口返回 "服务暂不可用"
- **Remotion 渲染失败**：捕获错误日志，定位到具体组件；向用户展示简化错误信息，提供 "重新生成" 或 "手动调整" 选项
- **磁盘空间不足**：素材导入、渲染前检查剩余空间；不足时阻止操作并提示清理

## 安全设计

### LLM API Key 存储
- 存储位置：`~/.easymotion/secrets.json`
- 加密方式：**AES-256-GCM**，密钥由系统钥匙串派生（macOS Keychain / Windows DPAPI / Linux libsecret）
- 内存安全：API Key 仅在需要发起请求时临时解密，使用后立即从内存清除（`crypto.randomFillSync` 覆写）
- 传输安全：所有 LLM 请求强制 HTTPS，禁止 HTTP 回退

### 文件沙箱
- 所有文件操作（读取/写入/删除）限定在项目根目录内，禁止访问系统敏感路径
- 子项目生成的 Remotion 代码在 iframe 中运行，iframe `sandbox` 属性限制为 `allow-scripts`（禁止 `allow-same-origin`，防止访问 Electron 内部 API）
- 素材导入时对 SVG 文件进行 XSS 扫描（禁止 `<script>`、事件处理器）

### 代码执行安全
- LLM 生成的 Remotion 代码在写入项目前，先通过 `esbuild` 语法校验（不执行，仅检查 AST）
- 禁止使用的 API 列表（正则白名单）：禁止 `require('fs')`、`require('child_process')`、`<iframe>`、`<object>`、`<embed>` 等
- 发现危险代码时标记为 "生成异常"，不写入项目，提示用户重新生成
