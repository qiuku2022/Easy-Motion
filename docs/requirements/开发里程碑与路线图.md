# 开发里程碑与路线图

## 概述

本文档定义 EasyMotion 从项目启动到 MVP 发布的开发路线图，包括里程碑划分、模块依赖关系、交付物定义和验收标准。

---

## 模块依赖图

```
┌─────────────────────────────────────────────────────────────────┐
│                        基础设施层 (Foundation)                    │
```

**依赖关系说明：**

- **基础设施层** 所有模块的基础，必须最先完成
- **数据与核心层** 依赖基础设施，为业务功能提供数据支持
- **业务功能层** 各模块相对独立，但 Generator 和 Remotion 预览必须配合工作
- **时间线 UI** 和 **Generator** 强耦合（UI 修改 → Generator 重新生成 → 预览更新）
- **对话面板** 和 **Generator** 强耦合（AI 生成 → 修改 timeline → Generator 触发）

## 里程碑划分

### M0：项目脚手架（Week 1-2）

**目标：** 搭建可运行的 monorepo 骨架，确保开发环境就绪

**交付物：**

1. **monorepo 目录结构**
  - `apps/electron/`：Electron 主应用（主进程 + 渲染进程 + 预加载脚本）
  - `apps/python/`：Python FastAPI 模块
  - `packages/shared/`：共享类型和常量
  - `pnpm-workspace.yaml`：workspace 配置
2. **Electron 最小可运行骨架**
  - 主进程入口（`main/index.ts`）
  - 预加载脚本（`preload/index.ts`）
  - 渲染进程入口（`renderer/main.tsx`）
  - 开发模式热重载配置
3. **Python FastAPI 最小可运行服务**
  - `main.py` FastAPI 入口
  - `src/api/` 路由目录
  - `src/crawler/` 爬虫目录
  - `src/data_processor/` 数据处理目录
4. **构建工具链**
  - Vite 配置（渲染进程开发服务器 + 打包）
  - TypeScript 配置（严格模式）
  - ESLint + Prettier 配置
  - 根目录 package.json 脚本（dev/build/test/lint）

**验收标准：**

- [ ] `pnpm install` 成功安装所有依赖
- [ ] `pnpm dev` 启动 Electron 窗口，显示 "Hello EasyMotion"
- [ ] Python FastAPI 服务可独立启动，`/health` 接口返回 200
- [ ] `pnpm build` 成功打包渲染进程和主进程
- [ ] `pnpm lint` 无错误
- [ ] CI（GitHub Actions）基础工作流通过（安装 + 构建 + lint）

**技术难点：**

- Electron 主进程和渲染进程的热重载配置（vite-plugin-electron）
- Python 子进程在开发模式下的自动启动和端口管理
- pnpm workspace 的依赖共享和版本锁定

### M1：项目系统 + 文件服务（Week 3-4）

**目标：** 实现项目的创建、打开、保存、删除，建立文件服务基础

**交付物：**

1. **主进程项目服务**
  - `project-service.ts`：项目 CRUD（新建/打开/保存/关闭/删除）
  - `file-service.ts`：文件系统抽象（复制/删除/重命名/校验）
  - IPC handlers：`main:project:`* 接口完整实现
2. **项目数据结构**
  - `project.json` 读写（含版本管理、自动保存）
  - `subproject.json` 读写
  - 默认项目模板生成（`templates/default-project/`）
3. **项目窗口 UI**
  - 独立项目窗口（`project-manager` 窗口类型）
  - 项目列表展示（缩略图、名称、时间）
  - 新建项目对话框（分辨率/帧率/时长/模板选择）
4. **文件服务基础设施**
  - 单写队列机制
  - 原子写入（临时文件 + 重命名）
  - SHA-256 校验 + 备份机制
  - `~/.easymotion/` 配置目录初始化

**验收标准：**

- [ ] 用户点击"新建项目" → 填写参数 → 项目创建成功 → 文件系统存在完整目录结构
- [ ] 项目列表正确显示所有项目，支持搜索
- [ ] 打开项目 → 加载 project.json → 进入主工作区
- [ ] 修改项目（如重命名子项目）→ 自动保存 → 关闭后重新打开数据完整
- [ ] 删除项目 → 可选保留输出文件 → 目录清理正确
- [ ] 文件写入异常时（磁盘满/权限不足）→ 错误提示 + 回退到备份

**技术难点：**

- 跨平台路径处理（Windows `C:\` vs macOS/Linux `/Users/`）
- 大文件复制时的进度反馈和取消机制
- 并发文件操作的安全性（单写队列 + 校验的实现）

### M2：时间线数据模型 + Generator（Week 5-7）

**目标：** 实现时间线的核心数据结构和代码生成器，建立"数据 → 代码 → 预览"的基础链路

**交付物：**

1. **时间线数据模型（TypeScript 类型）**
  - `Track`、`Clip`、`Keyframe`、`AnimationConfig` 等接口
  - `timelineStore`（Zustand + Immer）基础实现
  - 数据验证（zod schema 或自定义校验）
2. **Generator（核心模块）**
  - `generator/index.ts`：入口，接收 timeline JSON，输出 TSX 代码
  - `template-engine.ts`：模板引擎（ejs / handlebars / 字符串拼接）
  - `component-registry.ts`：Layer 组件注册表（text/image/video/shape/chart）
  - 生成 `Root.tsx` + `MainSequence.tsx` + Layer 组件引用
3. **基础 Layer 组件**
  - `TextLayer.tsx`：文字渲染（支持基础样式）
  - `ImageLayer.tsx`：图片渲染（`<Img>` 组件）
  - `ShapeLayer.tsx`：形状渲染（rect/circle）
  - `AudioLayer.tsx`：音频渲染（`<Audio>` 组件）
4. **基础时间线 UI**
  - 轨道列表显示（名称、类型、可见/锁定状态）
  - 片段在时间线上的显示（矩形块，显示名称和时长）
  - 播放头显示和基本控制（播放/暂停/seek）

**验收标准：**

- [ ] 手动创建 timeline JSON（含 2 个 text 轨道）→ Generator 生成 TSX → esbuild 编译通过
- [ ] 生成的 Remotion 代码在浏览器中可运行，显示正确的动画效果
- [ ] 修改 timeline（如改变 clip 位置）→ Generator 重新生成 → 代码更新
- [ ] UI 显示轨道列表，与 timeline 数据同步
- [ ] 播放头可拖拽，显示当前帧号

**技术难点：**

- Generator 的模板引擎选型（要支持嵌套循环、条件渲染）
- Layer 组件 Props 接口设计（通用性 vs 类型安全）
- 时间线 JSON 到 Remotion `Sequence` 的精确映射（时序、层级、叠加）

### M3：Remotion 预览集成（Week 8-9）

**目标：** 将 Remotion 预览嵌入 Electron 主窗口，实现"修改即预览"

**交付物：**

1. **iframe 集成**
  - 子项目 webpack dev server 启动和管理
  - iframe 嵌入 Remotion Player
  - iframe 与父窗口的 `postMessage` 通信
2. **预览控制**
  - 播放/暂停/seekTo 控制
  - 播放头位置同步（iframe → 父窗口 → 时间线 UI）
  - 预览缩放（50%-200%）
3. **实时重载**
  - Generator 生成代码 → 写入 `remotion/src/` → 触发 HMR
  - HMR 失败时的自动 reload 机制
  - 生成耗时指示器（"正在生成预览..."）
4. **预览窗口 UI**
  - 播放控制按钮（播放/暂停/逐帧/首尾跳转）
  - 时间码显示（帧号 / 总帧数）
  - 全屏预览模式

**验收标准：**

- [ ] 打开子项目 → iframe 加载 Remotion Player → 显示生成的动画
- [ ] 点击播放按钮 → 动画播放 → 播放头在时间线 UI 上同步移动
- [ ] 拖拽时间线播放头 → iframe 预览 seek 到对应帧
- [ ] 修改 timeline（如移动 clip）→ 500ms 后预览自动更新
- [ ] 点击 iframe 内元素 → 父窗口选中对应 clip → 右侧面板显示属性

**技术难点：**

- 子项目 webpack dev server 在 Electron 中的启动和管理（端口冲突、进程生命周期）
- iframe 与父窗口的跨域通信（`postMessage` 的安全性和可靠性）
- HMR 在 iframe 中的稳定性（webpack dev server 的配置）
- 复杂项目的预览性能（8+ 轨道时的帧率优化）

### M4：时间线编辑 UI（Week 10-12）

**目标：** 实现完整的时间线拖拽编辑功能

**交付物：**

1. **片段拖拽**
  - 拖拽片段调整位置（修改 `startInFrames`）
  - 拖拽片段边缘调整时长（修改 `durationInFrames`）
  - 拖拽素材到时间线创建新片段
2. **轨道操作**
  - 添加/删除轨道
  - 拖拽轨道调整层级（修改 `order`）
  - 轨道锁定/隐藏/独奏/静音
3. **吸附（Snap）**
  - 片段边界吸附
  - 播放头位置吸附
  - 自定义标记点吸附
  - 按住 Alt 临时禁用吸附
4. **时间线缩放**
  - 缩放控制（帧/秒/分钟视图）
  - 滚动和视口管理
5. **撤销/重做**
  - Command Pattern 实现
  - 撤销栈（内存，50 步）
  - 快捷键（Ctrl+Z / Ctrl+Y）

**验收标准：**

- [ ] 拖拽片段到新的时间位置 → timeline 数据更新 → 预览同步
- [ ] 拖拽片段边缘改变时长 → 预览中片段持续时间变化
- [ ] 拖拽素材到时间线 → 自动创建对应类型轨道和片段
- [ ] 片段靠近边界时自动吸附 → 松开鼠标后位置精确对齐
- [ ] 
- [ ] 删除片段 → 撤销（Ctrl+Z）→ 片段恢复
- [ ] 连续操作 10 次 → 可连续撤销 10 次 → 撤销栈满后最早的被丢弃

**技术难点：**

- 拖拽的精确像素计算（时间线坐标 ↔ 帧号的映射）
- 吸附算法的性能（大量片段时的边界检测）
- 撤销/重做与 Generator 的协调（undo 后重新生成代码）

### M5：AI 对话面板 + Agent 集成（Week 13-15）

**目标：** 实现对话式 AI 交互，用户可通过自然语言生成和调整动画

**交付物：**

1. **对话面板 UI**
  - 聊天界面（消息列表、输入框、图片上传）
  - 消息类型区分（用户 / AI / 系统通知）
  - 图片缩略图显示和重新排序
  - 收起/展开控制
2. **LLM 服务集成**
  - `llm-service.ts`：LLM API 代理（支持 OpenAI / Claude）
  - 流式响应处理（SSE / 流式 JSON）
  - API Key 加密存储和读取
  - 设置面板：LLM 提供商选择、API Key 输入、模型选择
3. **Agent 工作流**
  - 意图解析（Prompt + LLM 调用）
  - 工具调用链（`createTrack` → `createClip` → `updateClip`）
  - 视觉解析（多模态 LLM 图片分析）
  - 代码生成和校验（esbuild + 安全白名单）
4. **对话调整**
  - 参数调整解析（"字体大一点" → `fontSize * 1.2`）
  - 元素定位（`queryElement` 工具）
  - 冲突检测和提示（手动修改后 AI 调整时提示覆盖）

**验收标准：**

- [ ] 用户输入"创建一个标题写着 Hello" → AI 响应 → 时间线出现 text 轨道 → 预览显示 "Hello"
- [ ] 用户上传参考图片 + 输入需求 → AI 解析图片 → 生成与参考图布局相似的动画
- [ ] 用户输入"字体大一点" → AI 调整 fontSize → 预览文字变大
- [ ] LLM 调用超时 → 自动重试 1 次 → 仍超时进入简化模式 → 生成基础文字动画
- [ ] 生成代码含安全违规 → 拒绝写入 → 提示用户重新生成
- [ ] 对话历史保存到 `conversation.json`，切换子项目后历史独立

**技术难点：**

- LLM Prompt 工程（意图解析的准确率、视觉解析的稳定性）
- 流式响应的 UI 渲染（消息逐字显示、进度指示）
- 工具调用链的错误处理（中间步骤失败时的回滚）
- 多模态 LLM 的响应解析和异常处理

**技术难点：**

## 风险与应对


| 风险               | 概率  | 影响  | 应对策略                                   |
