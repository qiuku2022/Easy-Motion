# 数据备份与恢复

## 概述

本文档定义 EasyMotion 在数据备份、自动快照、手动导出和灾难恢复方面的策略与操作流程，确保用户项目数据不因软件故障、硬件损坏或误操作而丢失。

> **版本**：v0.1  
> **最后更新**：2026-05-30  
> **关联文档**：技术规格.md、非功能需求.md、IPC通信协议规范.md、数据库Schema设计.md

---

## 1. 数据分类与保护级别

| 数据类型 | 存储位置 | 重要性 | 备份策略 |
|----------|----------|--------|----------|
| 项目文件（`project.json`、`subproject.json`） | 项目目录 | 🔴 核心 | 自动快照 + 手动备份 |
| 素材原文件（`assets/`） | 项目目录 | 🔴 核心 | 手动备份（体积大） |
| 对话历史（`conversation.json`） | 子项目目录 | 🟡 高 | 自动快照 + 导出 |
| 素材元数据索引（`library.db`） | `~/EasyMotion/` | 🟡 高 | 每日自动备份 |
| 缩略图缓存（`.thumbnails/`） | 项目目录 | 🟢 低 | 不备份（可重建） |
| 应用设置（`settings.json`） | `~/.easymotion/` | 🟢 低 | 导出时包含 |
| 时间线快照（`timeline-snapshots/`） | 项目目录 | 🟡 中 | 自动轮转 |

## 2. 自动快照系统

### 2.1 快照触发条件

| 触发场景 | 说明 | 保留策略 |
|----------|------|----------|
| **定时快照** | 每 30 秒（若时间线有变更） | 保留最近 50 个 |
| **操作后快照** | 关键操作完成后立即生成 | 保留最近 20 个 |
| **生成动画完成** | Agent 生成代码写入后 | 保留最近 10 个 |
| **手动保存** | 用户按 `Ctrl+S` | 保留最近 10 个 |
| **会话结束** | 正常关闭应用前 | 保留最近 5 个 |

### 2.2 快照内容

```json
// timeline-snapshots/{timestamp}.json
{
  "snapshotId": "snap-1717065600000",
  "createdAt": 1717065600000,
  "trigger": "auto-save",         // auto-save | key-operation | generation | manual | session-end
  "subprojectId": "sub-001",
  "label": "",                    // 手动快照时用户可填写
  "data": {
    "timeline": { /* 完整 Timeline 对象 */ },
    "conversation": { /* 完整 Conversation 对象 */ },
    "version": "1.0"
  }
}
```

### 2.3 快照管理

- **数量限制**：每个子项目最多保留 50 个快照，超出时按 FIFO 删除最早的定时快照（保留手动快照和关键操作快照）。
- **大小控制**：单个快照文件大小通常 < 1MB（JSON 文本），50 个快照 < 50MB。
- **清理**：项目删除时，同步删除该项目的所有快照。

### 2.4 快照恢复流程

```
用户打开「历史版本」面板
    │
    ├── 显示快照列表（按时间倒序，显示触发原因和标签）
    │
    ├── 用户选中快照 → 点击「预览」
    │      └── 临时加载快照 timeline 到预览窗口（不覆盖当前）
    │
    ├── 用户点击「恢复到此版本」
    │      └── 确认弹窗："恢复后将丢失当前未保存的更改，是否继续？"
    │
    └── 确认后
           ├── 1. 备份当前 timeline 到临时快照（防止误恢复）
           ├── 2. 将选中快照写入 subproject.json
           ├── 3. 重新加载时间线
           ├── 4. 通知 Generator 重新生成代码
           └── 5. 刷新预览窗口
```

## 3. library.db 自动备份

### 3.1 备份策略

```
每日 02:00 自动执行
    │
    ├── 1. 检查 library.db 是否被占用
    │      └── 被占用 → 延迟到 02:30 重试，最多重试 3 次
    │
    ├── 2. 创建备份 copy
    │      └── ~/EasyMotion/backups/library.db.{日期}.bak
    │
    ├── 3. 压缩备份（gzip）
    │      └── library.db.{日期}.bak.gz
    │
    └── 4. 清理旧备份
           └── 保留最近 7 天 + 每月 1 号备份保留 3 个月
```

### 3.2 备份存储位置

```
~/EasyMotion/
├── library.db                      # 主数据库
└── backups/
    ├── library.db.2026-05-30.bak.gz
    ├── library.db.2026-05-29.bak.gz
    └── ...
```

### 3.3 数据库损坏恢复

```
启动时检测到 library.db 损坏（SQLite 返回 CORRUPT）
    │
    ├── 1. 提示用户："检测到素材库损坏，是否从备份恢复？"
    │
    ├── 2. 用户选择「恢复」
    │      ├── a. 列出可用备份（按日期倒序）
    │      ├── b. 用户选择备份版本
    │      ├── c. 关闭当前数据库连接
    │      ├── d. 用备份文件替换 library.db
    │      ├── e. 重新初始化连接
    │      └── f. 验证数据库完整性（PRAGMA integrity_check）
    │
    └── 3. 恢复失败或用户选择「不恢复」
           └── 创建新的空数据库，提示用户重新导入素材
```

## 4. 手动备份

### 4.1 项目备份（打包导出）

用户可将整个项目打包为 `.easymotion` 文件（ZIP 格式）：

```
「文件」→「导出项目备份」
    │
    ├── 用户选择保存路径
    │
    ├── 系统打包：
    │      ├── project.json
    │      ├── subprojects/
    │      │   └── {subproject}/
    │      │       ├── subproject.json
    │      │       ├── conversation.json
    │      │       └── remotion/
    │      ├── assets/
    │      │   ├── images/
    │      │   ├── videos/
    │      │   ├── audio/
    │      │   ├── data/
    │      │   └── fonts/
    │      ├── .thumbnails/          # 可选（用户勾选）
    │      └── timeline-snapshots/   # 可选（用户勾选）
    │
    └── 生成：{项目名}-{日期}.easymotion.zip
```

**打包选项：**
- ☑️ 包含素材原文件（默认勾选，如取消则仅导出引用）
- ☑️ 包含缩略图缓存（默认勾选）
- ☑️ 包含时间线快照（默认不勾选）
- ☑️ 包含设置（默认勾选）

### 4.2 增量备份（仅时间线）

针对大项目（素材 > 10GB），提供增量备份选项：

```
「文件」→「导出时间线」
    │
    └── 仅导出：
          ├── project.json
          ├── subproject.json
          ├── conversation.json
          └── assets-manifest.json（素材引用清单，不含原文件）
```

### 4.3 素材库备份

```
「文件」→「备份素材库」
    │
    ├── 导出 library.db
    ├── 导出素材元数据（JSON）
    └── 可选：打包所有素材原文件（生成 .zip）
```

## 5. 项目恢复

### 5.1 从备份文件恢复

```
「文件」→「从备份恢复项目」
    │
    ├── 用户选择 .easymotion.zip 文件
    │
    ├── 系统解压到临时目录
    │
    ├── 验证项目结构完整性
    │      └── 缺失关键文件 → 提示"备份文件已损坏"
    │
    ├── 检查素材完整性
    │      └── 缺失素材文件 → 提示"以下素材缺失，是否继续？"
    │
    ├── 用户选择恢复位置
    │      └── 如路径已存在同名项目 → 提示重命名或覆盖
    │
    └── 解压到目标位置，自动打开项目
```

### 5.2 跨设备迁移

```
设备 A：「导出项目备份」（含素材）
    │
    ├── 传输到设备 B（网盘/U盘/云同步）
    │
    └── 设备 B：「从备份恢复项目」
```

**注意事项：**
- 路径差异：恢复时自动修正素材绝对路径（以新项目路径为基准）。
- 字体素材：如使用了本地系统字体，跨设备可能缺失，需重新导入。

## 6. 灾难恢复流程

### 6.1 场景：项目目录被误删除

```
用户发现项目文件夹被删除
    │
    ├── 1. 检查 ~/.easymotion/last-projects.json（最近打开的项目索引）
    │
    ├── 2. 提示用户："项目目录已不存在，是否从最近快照恢复？"
    │
    ├── 3. 用户选择「恢复」
    │      ├── a. 扫描 timeline-snapshots/ 目录
    │      ├── b. 列出可用快照
    │      ├── c. 恢复 project.json + subproject.json
    │      └── d. 提示：素材文件需手动重新导入
    │
    └── 4. 恢复后检查 library.db 一致性
           └── 素材记录指向不存在的文件 → 标记为"素材缺失"
```

### 6.2 场景：应用崩溃后未保存数据

```
应用异常崩溃，用户重新启动
    │
    ├── 1. 启动时扫描所有项目的 timeline-snapshots/
    │
    ├── 2. 发现"未正常关闭"标记（崩溃前写入的标记文件）
    │
    ├── 3. 提示用户："检测到上次未正常退出，是否恢复到最近自动保存的版本？"
    │      ├── 显示：最近自动保存时间、当前文件修改时间
    │      └── 用户选择：「恢复」/「放弃」（加载磁盘当前版本）
    │
    └── 4. 恢复后删除"未正常关闭"标记
```

### 6.3 场景：硬盘损坏（部分数据丢失）

```
用户报告项目文件损坏或丢失
    │
    ├── 1. 检查项目目录完整性（project.json、subproject.json、assets/）
    │
    ├── 2. 如核心文件损坏：
    │      ├── 尝试从 timeline-snapshots/ 恢复最近版本
    │      └── 无法恢复 → 建议从 .easymotion 备份恢复
    │
    ├── 3. 如素材文件丢失：
    │      ├── 扫描 assets/ 目录与 library.db 记录对比
    │      ├── 列出缺失的素材
    │      └── 提示用户重新导入或从备份恢复
    │
    └── 4. 如 library.db 损坏：
           └── 执行「数据库损坏恢复」流程（见 3.3 节）
```

## 7. 备份存储建议

### 7.1 推荐用户策略

| 用户类型 | 推荐策略 | 频率 |
|----------|----------|------|
| **个人创作者** | 定期导出 .easymotion 备份到网盘 | 每周或每完成一个项目 |
| **团队/工作室** | 项目目录纳入版本控制或云同步 | 实时同步 |
| **重要商业项目** | 3-2-1 备份原则 | 每次关键里程碑 |

### 7.2 3-2-1 备份原则

- **3** 份数据副本（1 份原始 + 2 份备份）
- **2** 种不同存储介质（本地 + 云端/外部硬盘）
- **1** 份离线备份（不与计算机直接连接）

## 8. 导出格式规范

### 8.1 项目备份格式（.easymotion）

```
{项目名}-{日期}.easymotion.zip
├── manifest.json              # 备份元数据
│   {
│     "version": "1.0",
│     "backupDate": "2026-05-30T12:00:00Z",
│     "appVersion": "1.0.0",
│     "includesAssets": true,
│     "includesThumbnails": true,
│     "projectName": "Demo",
│     "projectId": "proj-001"
│   }
├── project.json
├── subprojects/
│   └── ...
├── assets/
│   └── ...
└── .thumbnails/               # 可选
```

### 8.2 时间线交换格式（.emt）

用于分享时间线配置（不含素材）：

```json
// {项目名}-{日期}.emt
{
  "version": "1.0",
  "type": "easymotion-timeline",
  "timeline": { /* Timeline 对象 */ },
  "conversation": { /* Conversation 对象（可选） */ },
  "assetManifest": [
    { "id": "asset-001", "name": "logo.png", "type": "image" }
    // 仅记录引用，不含文件
  ]
}
```

**导入 .emt 时：**
- 创建新项目或子项目。
- 提示用户按 assetManifest 重新导入缺失素材。
- 素材匹配规则：先按 `contentHash` 匹配，再按 `name` 匹配。

## 9. 数据清理策略

### 9.1 自动清理

| 数据 | 清理条件 | 保留期限 |
|------|----------|----------|
| 时间线快照 | 数量超过 50 | 永久（仅限制数量） |
| 缩略图缓存 | 总大小 > 500MB | LRU，最少保留最近 100 个 |
| 临时文件（`~/.easymotion/temp/`） | 文件年龄 > 24h | 无 |
| library.db 备份 | 数量超过策略 | 7 天 + 每月备份 3 个月 |
| 崩溃日志 | 数量超过 50 | 30 天 |

### 9.2 手动清理

设置面板 →「存储」→「清理缓存」：
- 清理缩略图缓存（保留最近 30 天）
- 清理临时文件
- 清理崩溃日志
- 压缩时间线快照（删除冗余中间版本）

## 10. 测试验证

| 测试项 | 方法 | 通过标准 |
|--------|------|----------|
| 自动快照生成 | 修改时间线后等待 30 秒 | `timeline-snapshots/` 出现新文件 |
| 快照恢复 | 恢复快照后对比 timeline | 与快照内容一致 |
| 数据库备份 | 检查 `~/EasyMotion/backups/` | 存在 .bak.gz 文件 |
| 项目打包 | 导出 .easymotion 并解压 | 结构完整，可重新打开 |
| 跨设备恢复 | 在另一台设备恢复备份 | 项目正常打开，素材可用 |
| 崩溃恢复 | 强制退出后重启 | 提示恢复最近自动保存版本 |
