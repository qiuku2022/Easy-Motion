# 测试策略

## 概述

本文档定义 EasyMotion 项目的分层测试策略，涵盖单元测试、集成测试、端到端测试和性能测试，确保各模块的质量和稳定性。

## 单元测试

### 1. Electron 主进程测试

**测试范围：** `apps/electron/src/main/` 下的纯逻辑模块

| 模块 | 测试重点 | 示例测试用例 |
|------|----------|-------------|
| `project-service.ts` | 项目 CRUD | 创建项目 → 验证目录结构；删除项目 → 验证清理 |
| `file-service.ts` | 文件操作 | 复制素材 → 验证目标路径；生成缩略图 → 验证输出文件 |
| `security-check.ts` | 安全校验 | 输入危险代码 → 验证拒绝；输入合法代码 → 验证通过 |
| `generator/` | 代码生成 | 输入 timeline JSON → 验证生成的 TSX 语法正确 |

**工具：** Vitest（Node.js 环境）
**Mock 策略：**
- `fs` 模块使用 `memfs` 替代真实文件系统
- Electron API（`app`、`dialog`）使用自定义 mock
- SQLite 使用内存数据库（`:memory:`）

**测试文件位置：** `apps/electron/src/main/__tests__/*.test.ts`

### 2. 渲染进程测试

**测试范围：** `apps/electron/src/renderer/src/` 下的 Hooks 和工具函数

| 模块 | 测试重点 | 示例测试用例 |
|------|----------|-------------|
| `stores/` | Store 逻辑 | timelineStore.addClip() → 验证状态变化；undo() → 验证回退 |
| `hooks/` | 自定义 Hooks | useKeyframeInterpolation → 验证插值计算正确 |
| `utils/` | 工具函数 | getValueByPath / setValueByPath → 验证嵌套路径解析 |

**工具：** Vitest + React Testing Library（UI 组件）
**Mock 策略：**
- Electron IPC 使用 `vitest.mock` mock
- Remotion API 使用简化 mock

**测试文件位置：** `apps/electron/src/renderer/src/**/__tests__/*.test.ts`

### 3. Python 模块测试

**测试范围：** `apps/python/src/` 下的爬虫和数据处理模块

| 模块 | 测试重点 | 示例测试用例 |
|------|----------|-------------|
| `crawler/` | 网页爬取 | 输入测试 URL → 验证返回内容结构；模拟反爬 → 验证重试逻辑 |
| `data_processor/` | 数据解析 | 输入测试 CSV → 验证 JSON 输出；输入异常数据 → 验证错误处理 |
| `api/` | FastAPI 路由 | POST /crawl → 验证响应格式；健康检查 → 验证状态码 |

**工具：** pytest + pytest-asyncio + httpx（测试 FastAPI）
**Mock 策略：**
- Playwright 使用 `pytest-playwright` 的同步 API mock
- HTTP 请求使用 `respx` 或 `responses` mock

**测试文件位置：** `apps/python/tests/test_*.py`

### 4. Remotion 组件测试

**测试范围：** `apps/electron/resources/templates/default-project/remotion/src/components/layers/`

| 组件 | 测试重点 | 示例测试用例 |
|------|----------|-------------|
| `TextLayer.tsx` | 渲染输出 | 传入 props → 验证 DOM 结构和样式 |
| `ImageLayer.tsx` | 加载行为 | 传入 src → 验证 Img 组件调用参数 |
| `ShapeLayer.tsx` | 几何计算 | 传入 shape="circle" → 验证 CSS 生成正确 |

**工具：** Vitest + React Testing Library + `remotion` 测试工具
**特殊处理：**
- `useCurrentFrame()` 使用 `remotion` 的测试 helper 设置模拟帧号
- 图片素材使用 base64 内联或 mock

**测试文件位置：** `apps/electron/resources/templates/default-project/remotion/src/components/__tests__/*.test.tsx`

## 集成测试

### 1. IPC 通信测试

**目标：** 验证主进程 ↔ 渲染进程的 IPC 接口契约

| 测试场景 | 验证内容 |
|----------|----------|
| `main:project:create` | 渲染进程发送请求 → 主进程创建项目 → 返回成功响应 |
| `renderer:project:loaded` | 主进程加载项目 → 广播事件 → 渲染进程接收并更新 Store |
| 错误传递 | 主进程操作失败 → 错误信息正确传递到渲染进程 |

**工具：** Vitest + 自定义 Electron IPC mock（模拟双向通信）

### 2. Generator 集成测试

**目标：** 验证 Timeline JSON → TSX 代码 → 可运行组件的完整链路

| 测试场景 | 验证内容 |
|----------|----------|
| 简单文字动画 | timeline 含 1 个 text clip → 生成代码 → esbuild 编译通过 → 渲染无错误 |
| 多轨道叠加 | timeline 含 3 个轨道 → 验证生成的 Sequence 嵌套顺序正确 |
| 关键帧动画 | timeline 含关键帧 → 验证生成的组件在指定帧号输出正确值 |

**工具：** Vitest + esbuild（语法校验）+ React Testing Library（渲染验证）

### 3. Python FastAPI 集成测试

**目标：** 验证 Node.js ↔ Python HTTP 通信

| 测试场景 | 验证内容 |
|----------|----------|
| 爬虫服务 | Node.js 调用 `POST http://localhost:8000/crawl` → 返回正确数据结构 |
| 错误处理 | Python 服务未启动 → Node.js 返回 "服务暂不可用" |
| 心跳检测 | Python 进程崩溃 → Node.js 自动重启 → 服务恢复 |

**工具：** pytest + `TestClient`(FastAPI) + `subprocess` 管理

## 端到端测试（E2E）

### 测试范围

验证完整的用户工作流：
1. 启动应用 → 打开项目窗口 → 创建项目
2. 输入文字需求 → AI 生成动画 → 预览效果
3. 对话调整 → 预览更新 → 导出视频

### 测试用例

| 用例编号 | 场景 | 步骤 | 预期结果 |
|----------|------|------|----------|
| E2E-001 | 新建项目工作流 | 1. 点击"新建项目"<br>2. 选择空白项目<br>3. 输入项目名称<br>4. 点击创建 | 项目创建成功，进入主窗口，显示默认时间线 |
| E2E-002 | AI 生成动画 | 1. 在对话面板输入"创建一个标题写着Hello"r>2. 点击发送<br>3. 等待生成完成 | 时间线出现新的 text 轨道，预览窗口显示"Hello"标题 |
| E2E-003 | 对话调整 | 1. 输入"字体大一点"<br>2. 等待 AI 响应 | 标题字体变大，预览窗口更新 |
| E2E-004 | 时间线拖拽 | 1. 拖拽片段改变位置<br>2. 观察预览 | 预览窗口播放头同步，片段位置变化反映在预览中 |
| E2E-005 | 素材导入 | 1. 拖拽图片到素材库<br>2. 拖拽图片到时间线 | 素材出现在素材库，时间线创建 image 轨道，预览显示图片 |
| E2E-006 | 导出视频 | 1. 点击"导出"<br>2. 选择 MP4 格式<br>3. 点击确认 | 显示渲染进度，完成后输出目录出现 MP4 文件 |
| E2E-007 | 撤销重做 | 1. 删除一个片段<br>2. 按 Ctrl+Z 撤销<br>3. 按 Ctrl+Y 重做 | 片段先消失，撤销后恢复，重做后再次消失 |
| E2E-008 | 项目切换 | 1. 创建子项目 A 并添加内容<br>2. 创建子项目 B<br>3. 切换回子项目 A | 子项目 A 的内容完整恢复，时间线和预览正确 |

### 工具与配置

**工具：** Playwright + `@playwright/test` + `electron` fixture

**配置：**
```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  fullyParallel: false,  // Electron 应用不并行
  workers: 1,
  use: {
    trace: 'on-first-retry',
  },
  projects: [
    {
      name: 'electron',
      use: {
        browserName: 'chromium',
        launchOptions: {
          executablePath: require('electron'),
          args: ['--disable-gpu', '--no-sandbox'],
        },
      },
    },
  ],
});
```

**注意事项：**
- E2E 测试使用打包后的应用（非开发模式），确保测试环境与用户环境一致
- 每个测试用例独立工作目录（`test.use({ tmpDir: ... })`），避免状态污染
- 测试后自动清理生成的项目文件
- CI 环境中使用 `--headed=false` 无头模式

## 性能测试

### 测试目标

| 指标 | 目标值 | 测试方法 |
|------|--------|----------|
| 应用启动时间 | < 5 秒（从点击图标到主窗口就绪） | 使用 `performance.now()` 记录 |
| 预览帧率 | 1080p 项目 ≥ 25fps | Remotion Player 内置帧率统计 |
| Generator 响应时间 | < 2 秒（单轨道简单项目） | 记录 timeline 修改到预览更新的时间 |
| 内存占用 | 正常使用 < 1GB；复杂项目 < 2GB | `process.memoryUsage()` 监控 |
| 导出渲染速度 | 1080p 30fps 项目 ≥ 实时速度（1秒视频 ≤ 1秒渲染） | 记录渲染耗时与视频时长比值 |

### 测试场景

1. **大项目测试**：32 轨道、每轨道 10 个片段、含关键帧和动画
2. **高分辨率测试**：4K (3840×2160) 项目预览和导出
3. **长时间项目测试**：10 分钟 (18000 帧) 项目的时间线操作响应
4. **内存泄漏测试**：连续创建/删除 100 个项目，监控内存变化

## 测试执行策略

### 开发阶段（本地）

```bash
# 运行主进程单元测试
pnpm test:main

# 运行渲染进程单元测试
pnpm test:renderer

# 运行 Python 测试
pnpm test:python

# 运行 Remotion 组件测试
pnpm test:remotion

# 运行所有单元测试
pnpm test:unit

# 运行 E2E 测试（需要打包后的应用）
pnpm build && pnpm test:e2e
```

### CI/CD 阶段

| 阶段 | 执行测试 | 目标 |
|------|----------|------|
| **Pre-commit** | ESLint + TypeScript 类型检查 + 相关单元测试 | 快速反馈，< 30 秒 |
| **PR 检查** | 全部单元测试 + Python 测试 | 验证代码变更不破坏现有功能 |
| **Nightly** | 全部单元测试 + E2E 测试 + 性能测试 | 验证完整工作流和性能基准 |
| **Release** | 全部测试 + 多平台打包验证 | 确保发布版本质量 |

### 覆盖率目标

| 模块 | 目标覆盖率 | 说明 |
|------|------------|------|
| 主进程服务层 | ≥ 80% | project-service、file-service 等核心逻辑 |
| 渲染进程 Store | ≥ 75% | 状态变更逻辑 |
| 渲染进程 Hooks | ≥ 70% | 复杂计算逻辑 |
| Python 模块 | ≥ 70% | 爬虫和数据处理 |
| Remotion Layer 组件 | ≥ 60% | 基础渲染和边界处理 |
| E2E | 核心场景全覆盖 | 8 个核心工作流用例 |

## 测试数据管理

### 测试项目模板

`e2e/fixtures/test-project/` 提供标准化测试项目：
- `project.json`：预配置的项目（1080p、30fps、300 帧）
- `subprojects/default/`：含基础 Remotion 结构的子项目
- `assets/`：测试用素材（PNG、MP4、CSV）

### Mock LLM 服务

E2E 测试中 Agent 生成使用 mock LLM：
- 不调用真实 LLM API（避免成本和不稳定性）
- 预置响应映射：特定输入 → 返回固定的 timeline JSON
- 响应延迟模拟：人工添加 100-500ms 延迟，模拟真实体验

```typescript
// e2e/mocks/llm-mock.ts
const mockResponses: Record<string, Timeline> = {
  "创建一个标题写着Hello": {
    tracks: [/* 预设的 text 轨道 */]
  },
  "字体大一点": {
    // 返回 fontSize 增大的 patch
  },
};
```
