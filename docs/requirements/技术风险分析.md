# 技术风险分析

## 概述

本文档系统性地识别 EasyMotion 项目的关键技术风险，评估每个风险的概率和影响等级，并制定具体的缓解措施和 Plan B。

> **实施状态（2026-06-26）**：M5 已采用 **LangChain 结构化工具**（修改 timeline JSON）替代「直接生成 TSX」为主路径；预览走动态 `MainSequence`，Generator 性能风险已降低。下文部分条目仍按原设计描述，Plan B 多数已部分落地。

---

## 风险评估矩阵

| 概率 \ 影响 | 低（可控） | 中（需应对） | 高（可能阻塞） |
|-------------|-----------|-------------|---------------|
| **高（>60%）** | P2 关注 | P1 缓解 | **P0 攻关** |
| **中（30%-60%）** | P3 监控 | P2 关注 | P1 缓解 |
| **低（<30%）** | P4 记录 | P3 监控 | P2 关注 |

## P0 风险（必须攻关，可能阻塞项目）

### R1：LLM 生成代码不可靠

**风险描述：**
LLM 生成的 Remotion 代码存在语法错误、类型不匹配、安全违规或逻辑错误，导致预览崩溃或渲染失败。多模态 LLM 对参考图片的视觉解析可能提取错误的布局信息。

**具体表现：**
- 代码语法错误：缺少 import、括号不匹配、属性名拼写错误
- 类型错误：给 string 属性传 number、给 object 传 string
- 安全违规：生成 `require('fs')`、`<iframe>` 等禁止 API
- 逻辑错误：关键帧插值计算错误、动画时长为负数、片段重叠
- 视觉解析偏差：图片中元素位置估计偏差 > 20%、颜色提取错误

**概率：高（>70%）**
**影响：高（核心功能不可用）**

**根因分析：**
1. LLM 本质上是概率模型，无法保证 100% 语法正确
2. Remotion API 复杂，LLM 训练数据中 Remotion 代码占比低
3. 视觉解析依赖 LLM 的图像理解能力，对细节（如精确像素位置）不敏感
4. 用户输入模糊时，LLM 的"猜测"可能与用户意图不符

**缓解措施：**
1. **多层校验**
   - Layer 1：LLM 生成后自校验（在 Prompt 中要求输出前检查语法）
   - Layer 2：`esbuild` AST 静态分析（语法 + 白名单）
   - Layer 3：TypeScript 类型检查（写入后 `tsc --noEmit`）
   - Layer 4：运行时错误边界（React ErrorBoundary 包裹每个 Layer 组件）
2. **预设兜底**
   - 为每种动画类型提供高质量的预设模板（经过人工验证）
   - LLM 生成失败时，使用预设模板生成基础动画
3. **Prompt 工程优化**
   - 提供详细的 Remotion API 文档作为上下文（RAG 检索增强）
   - 使用 few-shot prompting，提供正确的代码示例
   - 要求 LLM 输出 JSON 配置而非原始代码，由 Generator 转换为代码（减少 LLM 直接写代码的错误）
4. **视觉解析增强**
   - 图片预处理：提高分辨率、裁剪无关区域
   - 多轮验证：LLM 提取布局后，用代码生成布局预览图，与用户原图对比，偏差大时提示

**Plan B：**
- 如果 LLM 生成代码的可用率 < 80%，切换为"配置驱动模式"：
  - 用户通过对话选择动画类型和参数（不直接生成代码）
  - 系统从预设库中组合匹配的模板
  - LLM 仅用于参数调整（如"字体大一点"），不生成完整代码

**量化指标：**
- 目标：LLM 生成代码的一次成功率 ≥ 80%（语法正确 + 安全合规 + 可运行）
- 监控：记录每次生成的错误类型和频率，持续优化 Prompt

### R2：Remotion iframe 预览性能瓶颈

**风险描述：**
复杂项目（8+ 轨道、4K 分辨率、大量关键帧）在 iframe 中的 Remotion 预览帧率可能 < 10fps，导致用户体验极差（卡顿、延迟）。

**具体表现：**
- 时间线拖拽播放头时，预览画面延迟 1-2 秒才更新
- 播放动画时帧率不稳定，肉眼可见卡顿
- Generator 频繁触发 HMR 导致 iframe 反复重载
- 内存占用持续增长，长时间使用后 Electron 渲染进程崩溃

**概率：中（~50%）**
**影响：高（核心体验受损）**

**根因分析：**
1. Remotion Player 在 iframe 中运行，每次 timeline 修改都要重新编译 webpack
2. 复杂项目的组件树深度大，React 渲染开销高
3. 高分辨率素材（4K 图片/视频）的解码和渲染消耗 GPU/CPU
4. iframe 与父窗口的 `postMessage` 通信有额外开销
5. Generator 每次全量生成 TSX，webpack HMR 需要重新编译大量代码

**缓解措施：**
1. **Generator 增量生成**
   - 仅生成变更的组件文件，不覆盖未修改的文件
   - 使用 webpack 的 `module.hot.accept()` 实现细粒度 HMR
   - 复杂项目时降低 Generator 触发频率（防抖 1000ms）
2. **预览性能分级**
   - 自动检测项目复杂度（轨道数、分辨率、素材大小）
   - 复杂项目自动降低预览分辨率（100% → 75% → 50%）
   - 关闭非必要特效（阴影、模糊、混合模式）
3. **内存管理**
   - 限制 iframe 内同时加载的素材数量（懒加载未在当前帧显示的素材）
   - 定期清理 webpack dev server 的缓存
   - 长时间使用后提示用户重启预览（iframe reload）
4. **通信优化**
   - `postMessage` 批量发送（不每帧都发，改为 30ms 节流）
   - 播放头位置同步使用 requestAnimationFrame

**Plan B：**
- 如果 iframe 预览性能始终不达标，改为"离线渲染预览"模式：
  - 用户修改 timeline 后，后台用 Remotion CLI 渲染低质量预览视频（360p，快速预设）
  - 预览窗口播放预渲染的视频文件，而非实时 React 渲染
  - 延迟增加到 5-10 秒，但帧率稳定

**量化指标：**
- 目标：1080p 项目（≤8 轨道）预览帧率 ≥ 25fps
- 监控：记录每次 Generator 触发后的预览帧率和内存占用

## P1 风险（需要缓解，有应对方案）

### R3：Python Playwright 跨平台兼容性

**风险描述：**
Python 爬虫模块依赖 Playwright，而 Playwright 在不同操作系统上的浏览器二进制文件（Chromium/Firefox/WebKit）存在兼容性差异，可能导致 Windows 或 Linux 上无法启动。

**具体表现：**
- Windows 上 Playwright 安装后找不到浏览器可执行文件
- Linux 上缺少系统依赖库（如 `libnss3`、`libxss1`）导致启动失败
- macOS Apple Silicon 和 Intel 的浏览器二进制不同
- Playwright 版本升级后浏览器二进制不兼容

**概率：中（~40%）**
**影响：中（爬虫功能不可用，但核心功能不受影响）**

**根因分析：**
1. Playwright 的浏览器二进制文件体积大（~100MB），打包时可能遗漏
2. 不同操作系统的动态链接库依赖不同
3. Electron 打包后的应用环境与开发环境差异大

**缓解措施：**
1. **浏览器二进制管理**
   - 开发时：使用 Playwright 的 `playwright install` 自动下载
   - 打包时：将浏览器二进制复制到 `resources/python/` 目录
   - 运行时：通过环境变量 `PLAYWRIGHT_BROWSERS_PATH` 指向打包路径
2. **依赖库检查**
   - 应用启动时检查系统依赖库（Linux 上运行 `ldd` 检查）
   - 缺失时提示用户安装（提供一键安装脚本）
3. **降级方案**
   - Playwright 失败时，回退到 `requests` + `beautifulsoup4`（静态页面爬取）
   - 提示用户"动态页面渲染暂不可用，已使用基础爬取模式"

**Plan B：**
- 如果 Playwright 跨平台问题无法解决，将爬虫功能拆分为独立下载包：
  - 基础安装包不包含 Python 模块
  - 用户首次使用爬虫功能时，后台下载对应平台的 Python + Playwright 包
  - 减少基础安装包体积，同时保证功能可用

### R4：Electron 打包体积爆炸

**风险描述：**
Electron 本身（~180MB）+ Remotion 运行时 + Python + Playwright + FFmpeg 的总和可能超过 500MB，远超目标的 300MB，影响用户下载和安装体验。

**具体表现：**
- 安装包体积 > 500MB
- 安装时间 > 5 分钟（普通宽带）
- 磁盘占用 > 1GB（安装后）
- 自动更新下载时间 > 10 分钟

**概率：中（~50%）**
**影响：中（用户体验差，但不阻塞功能）**

**根因分析：**
1. Electron 自带 Chromium（~150MB）和 Node.js（~50MB），不可裁剪
2. Remotion 依赖 FFmpeg（~80MB）用于视频编码
3. Python + Playwright 浏览器二进制（~150MB）
4. 项目模板和素材（~20MB）

**缓解措施：**
1. **体积审计**
   - 使用 `electron-builder` 的 `--publish=never` 打包后分析体积组成
   - 使用 `du -sh` 或专用工具定位大体积目录
2. **Python 模块拆分**
   - 基础安装包不包含 Python 模块（仅 ~300MB）
   - Python 模块作为"可选组件"，首次使用时后台下载
   - 用户可在设置中手动触发下载
3. **FFmpeg 精简**
   - 使用 Remotion 的 `ffmpeg` 包（已优化体积）
   - 只保留必要的编码器（H.264、VP9），移除不用的解码器
4. **压缩优化**
   - 启用 `asar` 压缩（~20% 体积减少）
   - 使用 `upx` 压缩可执行文件（Windows/Linux）
   - 安装包使用 LZMA 压缩（NSIS 和 DMG 的默认压缩）

**Plan B：**
- 如果体积仍无法压缩到 300MB，接受 400MB 的现实，但提供"精简版"和"完整版"：
  - 精简版：不含 Python 模块，仅核心功能（~300MB）
  - 完整版：含所有功能（~450MB）

### R5：SQLite 并发写冲突

**风险描述：**
多个操作同时写 SQLite（素材导入、项目创建、元数据更新）可能导致数据库锁定或数据损坏。

**具体表现：**
- 素材批量导入时，SQLite 返回 `SQLITE_BUSY`
- 项目创建和素材导入同时进行，其中一个失败
- 数据库文件损坏，无法读取素材列表

**概率：中（~40%）**
**影响：中（数据损坏，但可恢复）**

**根因分析：**
1. SQLite 默认文件锁是排他的，写操作会阻塞其他写操作
2. Electron 多窗口可能同时发起写请求
3. 素材导入是异步批量操作，容易产生并发

**缓解措施：**
1. **连接池 + 队列**
   - 使用 `better-sqlite3`（同步 API，性能更好）
   - 主进程中维护单一 SQLite 连接，所有写操作通过队列串行执行
   - 读取操作可并发（SQLite 支持多读单写）
2. **事务包裹**
   - 批量导入素材时，使用 `BEGIN TRANSACTION ... COMMIT` 包裹
   - 事务失败时自动回滚，保证数据一致性
3. **WAL 模式**
   - 启用 SQLite 的 WAL（Write-Ahead Logging）模式
   - 提高并发性能，减少锁定冲突
4. **备份机制**
   - 每天自动备份 `library.db` 到 `library.db.bak`
   - 数据库损坏时自动从备份恢复

**Plan B：**
- 如果 SQLite 并发问题持续，迁移到嵌入式 NoSQL（如 `leveldb` 或 `lmdb`）：
  - 读写性能更高，无锁冲突
  - 但查询功能较弱（无 SQL），需要重写查询逻辑

## P2 风险（需要关注，有监控机制）

### R6：Remotion 版本升级不兼容

**风险描述：**
未来 Remotion 发布 v5 或重大更新时，API 可能发生 Breaking Change，导致现有 Generator 生成的代码无法运行。

**概率：低（~20%，但长期必然发生）**
**影响：高（所有现有项目可能失效）**

**缓解措施：**
1. **版本锁定**
   - `package.json` 中固定版本号（`"remotion": "4.0.269"`）
   - 子项目独立 `node_modules`，不影响其他子项目
2. **兼容性测试**
   - 升级前在测试子项目验证所有 Layer 组件
   - 使用 Remotion 的 `upgrade` 工具检查 Breaking Change
3. **迁移脚本**
   - 开发 `remotion-migrator` 工具，自动将 v4 代码转换为 v5
   - 升级时批量应用到所有项目模板

---

### R7：跨平台文件系统差异

**风险描述：**
Windows（`C:\Users\`）、macOS（`/Users/`）、Linux（`/home/`）的路径格式、权限模型、特殊字符处理存在差异，导致文件操作在某一平台上失败。

**概率：中（~40%）**
**影响：中（功能异常）**

**缓解措施：**
1. **统一路径处理**
   - 使用 Node.js 的 `path` 模块（`path.join`、`path.resolve`）
   - 禁止硬编码路径分隔符（`/` 或 `\\`）
2. **权限处理**
   - 文件操作前检查权限（`fs.access`）
   - 无权限时提示用户选择其他路径
3. **CI 多平台测试**
   - GitHub Actions 在 Windows、macOS、Linux 上分别运行测试
   - 文件操作相关测试必须覆盖三平台

---

### R8：多模态 LLM 成本过高

**风险描述：**
GPT-4V / Claude 3 Opus 等多模态模型调用成本较高（图片分析每次 $0.01-$0.05），高频使用可能导致用户 API Key 额度快速消耗。

**概率：中（~50%）**
**影响：中（用户流失）**

**缓解措施：**
1. **缓存机制**
   - 相同图片的视觉解析结果缓存到本地（`{imageHash}.json`）
   - 24 小时内重复上传同一张图片，直接读取缓存
2. **智能降级**
   - 用户未上传图片时，使用纯文字模式（不调用多模态 API）
   - 图片分辨率 < 500px 时，先缩小到 500px 再发送（降低 token 数）
3. **本地模型支持**
   - 支持 Ollama / LM Studio 本地运行多模态模型
   - 本地模型完全免费，适合隐私敏感用户
4. **成本透明**
   - 设置面板显示本月 API 调用次数和预估费用
   - 接近额度上限时提醒用户

---

### R9：Generator 全量生成导致 HMR 失效

**风险描述：**
Generator 每次修改 timeline 后全量生成 `MainSequence.tsx`（可能 500+ 行），webpack HMR 需要重新编译大量代码，导致 HMR 失效或 iframe 反复 reload。

**概率：中（~50%）**
**影响：中（预览更新延迟增加）**

**缓解措施：**
1. **组件拆分**
   - 将 `MainSequence.tsx` 拆分为多个子组件（每个 clip 一个组件文件）
   - 只重新生成变更的 clip 组件，未变更的组件保持不动
2. **JSON 驱动渲染**
   - 生成一个轻量的 `config.json` 而非完整 TSX
   - `MainSequence.tsx` 读取 `config.json` 动态渲染（类似 JSON → React 的映射）
   - 修改 timeline 时只更新 `config.json`，不触发 webpack 编译
3. **热更新策略**
   - 简单修改（属性值变更）：直接更新 `config.json`，iframe 内 React 状态驱动更新
   - 复杂修改（新增/删除轨道）：触发 Generator 重新生成 + iframe reload

**Plan B：**
- 如果 HMR 始终不稳定，改为显式刷新按钮：
  - 用户修改后，预览窗口显示"点击刷新查看最新效果"
  - 用户主动触发 iframe reload，避免自动重载导致的闪烁

## 风险监控机制

| 监控项 | 触发条件 | 自动应对 | 人工介入 |
|--------|----------|----------|----------|
| LLM 生成失败率 | 连续 3 次失败 | 切换简化模式 | 检查 Prompt 和日志 |
| 预览帧率 | < 15fps 持续 5 秒 | 自动降级分辨率 | 检查项目复杂度 |
| 内存占用 | > 1.5GB | 提示用户重启预览 | 排查内存泄漏 |
| SQLite 锁定 | > 3 秒 | 自动重试 | 检查并发操作 |
| 打包体积 | > 350MB | 告警，启动体积优化 | 审查依赖 |
